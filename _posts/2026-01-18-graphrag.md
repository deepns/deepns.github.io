Standard RAG retrieves chunks. GraphRAG retrieves relationships. That distinction matters more than it sounds when your questions span entities that appear in different documents, sections, or contexts.

The classic failure mode: "Which of our microservices depend on the payment service, and what were the known issues in Q3?" A chunk retriever might pull in a relevant chunk here and another there, but it has no way to connect *PaymentService → depends_on → AuthService* from your architecture doc with *AuthService → had_incident → 2025-09-14* from your incident log. GraphRAG keeps those relationships explicit.

## The Limitation of Chunk-Level Retrieval

When you embed a document chunk, you encode the semantic content of that chunk in isolation. Cross-document entity relationships — the fact that "Deepak Sharma" in document A and "D. Sharma, CTO" in document B refer to the same person — don't survive chunking. Questions that require connecting information across documents fall flat.

This isn't a retrieval quality problem. It's a structural problem. The information exists in your corpus, but the data model doesn't preserve the connections.

## How GraphRAG Works

Microsoft's GraphRAG (open-sourced in 2024) addresses this with a two-phase approach:

**Indexing phase**: An LLM extracts entities and relationships from each chunk. These are merged into a knowledge graph. Communities of related entities are detected and summarized.

**Query phase**: Queries are answered by traversing the graph (global questions about community summaries) or combining graph context with vector retrieval (local questions about specific entities).

The extraction step looks roughly like this conceptually:

```
Chunk: "The AuthService, maintained by the Platform team, had three outages in Q3. 
        It depends on the TokenDB and the PaymentService for transaction validation."

Extracted entities: AuthService, Platform team, TokenDB, PaymentService
Extracted relationships:
  - AuthService --maintained_by--> Platform team
  - AuthService --depends_on--> TokenDB  
  - AuthService --depends_on--> PaymentService
  - AuthService --had_outages--> Q3 (count: 3)
```

## Building a Simple Entity Graph with networkx

You don't need Microsoft's full pipeline to get started. Here's a minimal implementation that extracts entities from chunks and builds a traversable graph:

```python
import networkx as nx
from openai import OpenAI
import json

client = OpenAI()

def extract_entities(chunk: str) -> list[dict]:
    response = client.chat.completions.create(
        model="gpt-4o-mini",
        response_format={"type": "json_object"},
        messages=[{
            "role": "user",
            "content": f"""Extract entities and relationships from this text.
Return JSON with keys "entities" (list of strings) and "relationships" 
(list of {"source", "relation", "target"} objects).

Text: {chunk}"""
        }]
    )
    return json.loads(response.choices[0].message.content)

def build_graph(chunks: list[str]) -> nx.DiGraph:
    G = nx.DiGraph()
    for chunk_id, chunk in enumerate(chunks):
        extracted = extract_entities(chunk)
        for entity in extracted.get("entities", []):
            if not G.has_node(entity):
                G.add_node(entity, chunks=[])
            G.nodes[entity]["chunks"].append(chunk_id)
        for rel in extracted.get("relationships", []):
            G.add_edge(
                rel["source"], rel["target"],
                relation=rel["relation"],
                chunk_id=chunk_id,
            )
    return G

def get_entity_context(G: nx.DiGraph, entity: str, depth: int = 2) -> list[str]:
    """Get all entities within N hops of the target entity."""
    if entity not in G:
        return []
    subgraph_nodes = nx.single_source_shortest_path_length(G, entity, cutoff=depth)
    context_chunks = []
    for node in subgraph_nodes:
        context_chunks.extend(G.nodes[node].get("chunks", []))
    return list(set(context_chunks))
```

## Query Decomposition: Graph + Vector

For a question like "What services depend on PaymentService and were they affected by the Q3 incidents?", the flow is:

1. Extract the anchor entity: `PaymentService`
2. Traverse the graph: find all nodes with `depends_on → PaymentService`
3. Collect chunk IDs associated with those neighbor nodes
4. Combine those graph-retrieved chunks with vector retrieval on the query
5. Feed the combined context to the LLM

```python
def graphrag_retrieve(query: str, G: nx.DiGraph, chunks: list[str], 
                       vector_retrieve_fn, top_k: int = 5) -> str:
    # Step 1: Find anchor entities in the query (simplified)
    entities_in_query = [node for node in G.nodes if node.lower() in query.lower()]

    # Step 2: Graph traversal
    graph_chunk_ids = set()
    for entity in entities_in_query:
        graph_chunk_ids.update(get_entity_context(G, entity, depth=2))

    graph_chunks = [chunks[i] for i in graph_chunk_ids if i < len(chunks)]

    # Step 3: Vector retrieval
    vector_chunks = vector_retrieve_fn(query, top_k=top_k)

    # Step 4: Merge and deduplicate
    all_context = list(set(graph_chunks + vector_chunks))
    return "\n\n".join(all_context)
```

## Honest Complexity and Cost Tradeoffs

GraphRAG is not free. The extraction LLM calls during indexing are expensive — for a large corpus, you're looking at real API spend before you answer a single query. Incremental updates (adding new documents without rebuilding the whole graph) require careful bookkeeping.

Entity resolution is the hard part. "PaymentService," "payment-svc," and "the payment microservice" need to merge into one graph node or you get a fragmented graph. This requires either a fuzzy matching pass or another LLM call per extraction batch.

My take: GraphRAG earns its complexity when your documents are deeply interconnected — knowledge bases, codebases with many files, research corpora with shared concepts. For a simple FAQ or document Q&A, standard RAG is almost certainly enough. Start there, and reach for the graph when you hit multi-hop questions that retrieval keeps failing on.
